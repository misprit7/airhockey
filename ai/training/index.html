Air Hockey RL Training System

Training a robotic air hockey agent in simulation using TD-MPC2 (model-based RL with learned world model + MPPI planning), then transferring to a physical cable-driven parallel robot. Two-phase pipeline: pretrain with --simple (idle opponent, original rewards), then self-play with --resume (Elo-rated pool, reward annealing to goals-only).

Why TD-MPC2?

Model-based RL learns a world model (dynamics + reward predictor) and uses MPPI planning at inference. This gives better sample efficiency than model-free methods (SAC/PPO) and natural multi-step lookahead for trajectory planning.

Two-Phase Pipeline

Phase 1 (Pretrain): --simple mode trains against an idle opponent with rich shaping rewards. Builds foundational hitting, aiming, and scoring skills. Phase 2 (Self-Play): --resume loads the pretrained agent, starts an Elo pool, and anneals shaping rewards to goals-only over 500k steps.

python bin/train_tdmpc2_fast.py --simple --steps 2000000
python bin/train_tdmpc2_fast.py --resume runs/simple/agent.pt --steps 5000000

Reward Design & Training Phases

Training uses a two-phase pipeline: pretrain against an idle opponent with rich shaping rewards (--simple), then self-play with reward annealing (--resume). A 4-stage curriculum (--curriculum) is also available in the code but not recommended — the simple two-phase approach consistently produces better results.

Puck-Stuck Reset

If the puck speed stays below 0.05 for 120 consecutive steps (2 seconds), the puck is reset to center with a random direction. Prevents stalled episodes.

Fixed Benchmark Eval

Every 500k steps, the agent plays 5 games each against idle, goalie, and follower opponents. A composite score tracks progress consistently.

Step-Based Metrics

All metrics tracked per 10k steps: goals, contacts, win rate, episodes. Per-episode logging removed to reduce noise.

4-Stage Curriculum (--curriculum) Available, Not Recommended

A 4-stage auto-advancing curriculum is implemented in the codebase but never produced better results than the simple two-phase pipeline. The added complexity of stage transitions, plateau detection, and reward schedule changes introduced more failure modes than it solved. Documented here for reference.

TD-MPC2 Architecture

World Model

TD-MPC2 learns a latent-space world model with 5 heads: encoder, dynamics (MLP), reward, termination, and policy. Q-values are estimated by a 5-member ensemble. The MLP dynamics model predicts the next latent state from the current state and action — simple, fast, and 14× lower prediction error than GRU (0.0009 vs 0.013).

MLP Dynamics — Simple Beats Recurrent

The dynamics model uses a standard feedforward MLP. GRU-based dynamics was tested and reverted — the MLP achieves 14× lower prediction error (0.0009 vs 0.013) while being simpler and faster. Velocities are provided directly in the observation, so temporal memory is unnecessary.

ComponentDetail

ArchitectureNormedLinear layers + SimNorm

Why MLP over GRU? With explicit velocities in the 14-dim observation, the encoder already captures full state information. The GRU's hidden state added complexity (per-env tracking, MPPI branching, episode boundary resets) without improving prediction quality — it actually hurt it by 14×.

Why not Transformer? Also tested and removed. At our scale (512-dim latent, 2-dim action), MLP is faster than self-attention for horizons ≤30.

  z_next = sim_norm(z_next)              # SimNorm → next latent z'

MPPI Planning (Batched)

At inference, MPPI samples 512 action trajectories over a 5-step horizon with 6 refinement iterations, evaluates them using the world model's Q-values, and selects the best action. This runs in parallel across all 64 environments in a single batched forward pass.

Animated: gray = sampled trajectories, colored = elite trajectories, green = selected action

Prioritized Experience Replay (PER)

The replay buffer uses prioritized sampling based on TD-error magnitude. High-error transitions (surprising outcomes) are replayed more frequently, accelerating learning on difficult scenarios.

Priority exponent (α)0.6
IS exponent (β)0.4 → 1.0
β annealingLinear over training
Min priority (ε)1e-6

β annealing corrects for the bias introduced by non-uniform sampling. It starts at 0.4 (allowing biased but fast early learning) and linearly increases to 1.0 (fully unbiased) by the end of training, ensuring convergence.

Training Loop

for step in range(total_steps):
  # 1. Plan actions (batched MPPI)
  actions = agent.act(obs, t0=reset_mask)

  # 2. Step all N envs
  obs, rewards, done = batch_env.step(actions)

  # 3. Shape rewards (curriculum stage)
  shaped = reward_shaper.compute(obs, rewards)

  buffer.add(trajectories)

  # 5. Gradient update (compiled _update)
  batch, weights = buffer.sample(prioritized=True)

FPS = environment frames per second (env steps × n_envs / wall time). Measured on RTX 5080. Self-play is ~2.5x slower due to opponent also running full MPPI planning.

Batched MPPI

Instead of planning separately for each env, MPPI trajectories for all 64 envs are evaluated in a single batched forward pass through the world model. This amortizes GPU kernel launch overhead.

torch.compile

torch.compile on _update, rollout (fused dynamics+reward), encoder, and Q-ensemble. Gives 1.9-3.4x speedup with ~33s warmup. Not applied to planning (graph breaks from randn/topk).

Fused Dynamics+Reward

The _rollout_dynamics_reward function unrolls dynamics and reward in a single compiled loop with shared cat (torch.cat([z, a]) computed once per step). Combined with compiled encoder and Q-ensemble, this fuses multiple GPU kernels into fewer launches.

Why Explicit Velocities Over Frame Stacking?

Frame stacking (5 position-only frames = 32 dims) was tested but reverted in favor of explicit velocities (14 dims). Direct velocity observations proved critical for learning quality — the world model error dropped 14× with MLP dynamics on velocity observations vs GRU dynamics on stacked frames. The simpler representation also makes the observation space smaller, the encoder's job easier, and enables a stateless MLP dynamics model.

Self-Play System

Self-play is the second phase of training, resuming from a pretrained agent. It uses an Elo-rated opponent pool to prevent cyclical forgetting and promote continual improvement. Current run (v20): 400k steps, 80% win rate, Elo 1161, scoring 29 goals/10k steps at 74 FPS. Latest recording: 7-6 competitive game.

Opponent Pool

Up to 100 snapshots of past agent checkpoints are stored on CPU. New snapshots are added every 50k steps. Old snapshots are pruned by evicting the one with the lowest match quality (furthest Elo from current agent). Works well with 40+ snapshots.

Sampling: Opponents are selected based on match quality (Elo proximity), with 2% exploration floor and 10% minimum probability for the most recent snapshot.

Inference: Opponents use full MPPI planning, same as the training agent. This ensures realistic opponent behavior and proper competitive pressure (74 FPS vs 185 FPS pretrain due to dual-agent planning).

Elo Rating Distribution

Each snapshot tracks win/loss statistics. Elo ratings approximate relative skill levels.

v20 pool: current agent Elo 1161. Older snapshots at lower Elo, recent ones approaching the current agent's skill.

Self-Play Reward Annealing

Self-play starts with the same rewards the agent was pretrained with (proximity=0.1, contact=5, directed_hit=2, progress=3, goals=+100/-5). This is critical — the world model's reward head learned these weights during pretrain, so MPPI planning depends on them matching. Over 500k steps, auxiliary shaping rewards are linearly annealed to zero, leaving only goal rewards. This smooth transition avoids the reward mismatch that broke planning in v19.

Why Self-Play?

Scripted opponents have a skill ceiling. Once the agent beats the follower consistently, there's nothing harder to train against. Self-play creates an endlessly escalating challenge where the opponent improves alongside the agent. Score handicaps (30% of episodes start with non-zero scores) prevent the agent from only learning to play from 0-0.

Game Recording & Match Tool

Periodically during training, a synchronous recording is run on the main thread. Recordings are saved in columnar JSON and viewable in the web UI. Can also record any checkpoint vs any checkpoint for direct comparison (e.g., match_latest_vs_early.json showed 6-0). Puck-stuck reset: if puck speed stays below 0.05 for 120 steps (2 seconds), the puck resets to center with a random direction.

Lessons Learned

Key failures from earlier training iterations and the fixes applied.

Shaping Rewards Dominated Goals

Problem: With higher shaping weights (contact 1.0, directed 0.5, placement 0.5) and lower goal rewards (+80/-40), auxiliary rewards contributed more total reward than goals. The agent optimized for hitting and shaping the puck rather than actually scoring.

Fix: Cut all shaping weights drastically (contact 0.1, directed 0.1, defense 0.1, progress 0.1, placement 0.2). Removed proximity entirely. Added linear reward annealing — shaping decays to 0 in the last 40% of each stage. Added contact cap (max 5 per episode) and penalty ramp (0→full over first 30%). Boosted goal reward to +160/-20 so goals dominate the signal.

Agent Rushing Without Scoring

Problem: The agent advanced stages by plateauing on shaping rewards without ever learning to score goals. It could chase and hit the puck consistently but couldn't convert hits into goals.

Fix: The min goals gate (1+ goals/ep) blocks advancement until scoring ability is proven. Combined with reduced shaping weights, goal rewards now dominate the signal. The Corner stage was removed entirely — it added training time without teaching useful skills beyond what Goalie provides.

Mirror Action Bug (Critical)

Problem: Opponent y-action was inverted when mirroring to the opponent's coordinate frame. All previous self-play opponents were playing with inverted vertical controls, making them dramatically weaker than they should have been. Win rates were inflated.

Fix: Fixed mirror_action_to_opponent to correctly negate the y-action. This was a critical bug that invalidated all prior self-play results.

Reward Mismatch Broke Planning (v19)

Problem: Self-play v19 switched to different reward weights than pretrain used. The world model's reward head had learned the pretrain rewards, so MPPI planning used wrong reward predictions. The agent's play quality collapsed.

Fix: Self-play now starts with the exact same rewards as pretrain and anneals shaping rewards to zero over 500k steps, giving the reward head time to adapt.

No Fixed Benchmark

Problem: Progress was only measured by within-stage reward curves, which made it hard to compare across stages or detect regressions. Reward scale changes between stages masked actual skill degradation.

Fix: Added fixed benchmark evaluation every 500k steps. The agent plays 5 games each against idle, goalie, and follower opponents with a composite score, providing a consistent skill metric across all stages.

Self-Play Fragility

Problem: The agent learned to play well from a 0-0 start but collapsed when behind. In real games, falling behind early led to increasingly poor play as the agent had no experience recovering from a deficit.

Fix: Added score handicaps in self-play: 30% of episodes start with non-zero scores (random advantage/disadvantage). This forces the agent to learn comeback strategies and to maintain composure when ahead.

GRU and Transformer Dynamics: Both Worse Than MLP

Both tested and reverted. GRU dynamics had 14× higher world model error (0.013 vs 0.0009) compared to a simple MLP. With explicit velocities in the observation, temporal memory is unnecessary — the full state is available at every timestep. Transformer-based dynamics was also benchmarked but at our scale (512-dim latent, 2-dim action), self-attention overhead far exceeds any benefit for horizons ≤30. Simple MLP wins on both accuracy and speed.

Sequential Env Stepping: 5x Slower

The original TD-MPC2 codebase stepped environments sequentially (one env at a time). Batching all 64 envs into vectorized NumPy operations gave a 5× speedup on the env stepping alone. Combined with batched MPPI (one GPU forward pass for all envs), this was the single largest throughput improvement.
